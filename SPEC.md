# SPEC.md

## 1. Problem Statement

Build a REST API service for ingesting virtual machine metric samples from a cloud platform. Each VM reports CPU, memory, disk, network ingress, and network egress metrics; the service accepts these samples quickly, processes them asynchronously, stores raw and derived data durably, and exposes query APIs for recent metrics, VM health, and active alerts. Alerting is based on CPU or memory exceeding configured thresholds for 3 consecutive samples. Correctness means accepted samples are persisted once, duplicate samples are not double-counted, derived aggregates and alert state are eventually updated, and query APIs return consistent PostgreSQL-backed state.

## 2. Requirements and Scale Inputs

Functional requirements:

- Ingest VM metric samples.
- Process samples asynchronously.
- Persist raw metric samples for 7 days.
- Persist hourly aggregates for 90 days.
- Query recent metrics for a VM over a time range.
- Query latest health status for a VM.
- Query active alerts.
- Mark a VM alerting when CPU or memory is above threshold for 3 consecutive samples.
- Handle client retries without double-counting duplicate metric samples.
- Retry processing failures.
- Send permanently failed messages to a DLQ in production.
- Provide tested, documented, containerized deployment to DigitalOcean.

Scale and non-functional requirements:

- 100,000 VMs.
- 5 metric types per VM.
- Each VM sends metrics every 10 seconds.
- Steady-state ingestion: about 50,000 samples/sec.
- Peak ingestion: about 100,000 samples/sec.
- Ingestion API p95 latency under 200ms.
- Processing is eventually consistent.
- Samples should usually be processed within 30 seconds.
- Deployment target: DigitalOcean App Platform, DigitalOcean Managed PostgreSQL, DigitalOcean Managed Kafka, App Platform Workers.

Scale impact:

- API must accept batches, not one request per sample. With 500-1,000 samples per request, steady-state is roughly 50-100 requests/sec instead of 50,000 requests/sec.
- Kafka is justified in the MVP because ingestion must be fast, asynchronous, buffered, retryable, and resilient to processing slowdowns.
- Workers should consume Kafka messages in a consumer group and scale horizontally by partition.
- Kafka partition key should be `vm_id` so per-VM metric ordering is more likely within a partition.
- PostgreSQL tables need indexes by `vm_id`, `metric_type`, `observed_at`, `status`, and idempotency keys.
- Raw sample retention should be implemented with time partitioning and partition drops in production; the interview MVP can document or stub the cleanup job.
- Query endpoints must be paginated and bounded.
- Observability should include ingestion latency, Kafka publish failures, consumer lag, worker processing latency, duplicate counts, DB latency, and alert counts.

## 4. Assumptions

- Producers send batches to reduce request QPS.
- Each sample includes a stable `sample_id` generated by the producer. If that is not available, the fallback natural idempotency key is `vm_id + metric_type + observed_at`.
- The MVP accepts the 5 metric types listed in the prompt only.
- `cpu_usage_percent`, `memory_usage_percent`, and `disk_usage_percent` are percentages in `[0, 100]`.
- `network_in_bytes` and `network_out_bytes` are non-negative byte counts for the reporting interval.
- Timestamps are ISO 8601 UTC and represent when the metric was observed.
- Ingestion is asynchronous: the API validates and enqueues samples to Kafka, then returns `202 Accepted`.
- PostgreSQL is the durable source of truth for queryable raw samples, hourly aggregates, alert state, and ingestion batch status.
- Kafka is part of the MVP because scale, latency, retry, and async requirements justify it.
- DigitalOcean App Platform ingress/load balancing is the external ingress layer.
- DigitalOcean Managed Kafka is used in production; local development may use Kafka in `docker-compose`.
- App Platform worker components run the Kafka consumers.
- Query endpoints use pagination with a default limit of 100 and a maximum limit of 1,000.
- Duplicate samples are ignored for processing and not double-counted.
- Duplicate batch submissions with the same `Idempotency-Key` return the existing ingestion batch response with `idempotent_replay=true`.
- Alert thresholds default to environment variables: `CPU_ALERT_THRESHOLD_PERCENT=90` and `MEMORY_ALERT_THRESHOLD_PERCENT=90`.
- Authentication, tenants, and per-VM threshold management are out of scope for the timed MVP unless the interviewer requires them.

## 7. Architecture Overview

### MVP Architecture

Async MVP:

Client
-> DigitalOcean App Platform ingress/load balancing
-> FastAPI web service
-> DigitalOcean Managed Kafka
-> App Platform worker
-> DigitalOcean Managed PostgreSQL

The FastAPI service validates batch requests, records an ingestion batch when an idempotency key is provided, publishes metric sample messages to Kafka, and returns `202 Accepted` after Kafka accepts the messages. Worker processes consume Kafka messages, insert raw samples into PostgreSQL with idempotent `ON CONFLICT DO NOTHING` behavior, update hourly aggregates, and update VM health and alerts.

For a 3-hour interview implementation, tests should exercise the publisher through an interface with a fake producer, and worker business logic should be testable without requiring a real Kafka broker in every test.

### Production Architecture

Production evolution:

- Multiple stateless FastAPI web replicas behind App Platform ingress/load balancing.
- DigitalOcean Managed PostgreSQL with connection pooling, table partitioning by `observed_at`, and retention jobs.
- DigitalOcean Managed Kafka with enough partitions to support worker parallelism.
- App Platform worker replicas in a shared consumer group.
- Retry topic or retry scheduling plus DLQ topic.
- Transactional outbox if the API must atomically record acceptance in PostgreSQL and publish to Kafka.
- Structured logs with request IDs and worker message IDs.
- Metrics and alerts for API latency, publish failures, consumer lag, worker failures, DB latency, and DLQ growth.
- Rate limiting and authentication/authorization.
- Schema migrations.
- Load testing to validate 50k-100k samples/sec assumptions.

## 8. API Contract

### GET /healthz

Purpose: liveness check.

Response `200`:

```json
{"status":"ok"}
```

Error cases: none expected unless the process is unhealthy.

### GET /readyz

Purpose: readiness check for serving traffic.

Response `200`:

```json
{"status":"ready","dependencies":{"database":"ok","kafka":"ok"}}
```

Error response `503` if PostgreSQL or Kafka is unavailable.

### POST /metric-samples

Purpose: ingest a batch of metric samples asynchronously.

Headers:

- `Idempotency-Key`: optional but recommended for safe batch retries.
- `X-Request-ID`: optional request correlation ID.

Request:

```json
{
  "samples": [
    {
      "sample_id": "sample-01JZ...",
      "vm_id": "vm-123",
      "metric_type": "cpu_usage_percent",
      "value": 91.5,
      "observed_at": "2026-06-21T21:00:00Z"
    }
  ]
}
```

Response `202`:

```json
{
  "batch_id": "batch-01JZ...",
  "accepted_count": 1,
  "idempotent_replay": false,
  "status_url": "/ingestion-batches/batch-01JZ..."
}
```

Duplicate idempotency-key response `200`:

```json
{
  "batch_id": "batch-01JZ...",
  "accepted_count": 1,
  "idempotent_replay": true,
  "status_url": "/ingestion-batches/batch-01JZ..."
}
```

Error cases:

- `400 bad_request`: empty batch or batch exceeds max size.
- `422 validation_error`: invalid field shape or unsupported metric type.
- `503 dependency_unavailable`: Kafka publish unavailable.
- `500 internal_server_error`: unexpected failure.

Purpose: query async ingestion status.

Response `200`:

- `404 not_found`: batch does not exist.

## 9. Data Model

Fields:

- `id`: UUID or ULID primary key.
- `idempotency_key`: text, nullable.
- `request_id`: text, nullable.
- `status`: enum, `ACCEPTED`, `PROCESSING`, `PROCESSED`, `PARTIAL_FAILED`, `FAILED`.
- `submitted_count`: integer, required.
- `processed_count`: integer, default 0.
- `duplicate_count`: integer, default 0.
- `failed_count`: integer, default 0.
- `last_error`: text, nullable.
- `created_at`: timestamptz.
- `updated_at`: timestamptz.

Constraints and indexes:

- Unique `idempotency_key` where not null.
- Index on `status`.
- Index on `created_at`.

### metric_samples

Fields:

- `id`: bigserial primary key.
- `sample_id`: text, required.
- `batch_id`: UUID or ULID, nullable relation to `ingestion_batches`.
- `vm_id`: text, required.
- `metric_type`: enum, required.
- `value`: numeric/double precision, required.
- `observed_at`: timestamptz, required.
- `received_at`: timestamptz, required.
- `processed_at`: timestamptz, nullable.
- `created_at`: timestamptz.

Constraints and indexes:

- Unique `sample_id`.
- Optional fallback unique constraint on `vm_id, metric_type, observed_at` if sample IDs are not available.
- Index on `vm_id, observed_at DESC`.
- Index on `vm_id, metric_type, observed_at DESC`.
- Index on `observed_at` for retention.
- Production: partition by `observed_at` using daily or hourly partitions.

### metric_hourly_aggregates

Fields:

- `id`: bigserial primary key.
- `vm_id`: text, required.
- `metric_type`: enum, required.
- `hour_start`: timestamptz, required.
- `sample_count`: integer, required.
- `sum_value`: double precision, required.
- `min_value`: double precision, required.
- `max_value`: double precision, required.
- `avg_value`: derived in query or stored.
- `updated_at`: timestamptz.

Constraints and indexes:

- Unique `vm_id, metric_type, hour_start`.
- Index on `vm_id, metric_type, hour_start DESC`.
- Index on `hour_start` for retention.

### vm_metric_health

Fields:

- `vm_id`: text, required.
- `metric_type`: enum, required for CPU and memory.
- `latest_value`: double precision, required.
- `latest_sample_id`: text, required.
- `latest_observed_at`: timestamptz, required.
- `threshold`: double precision, required.
- `consecutive_breaches`: integer, required.
- `alerting`: boolean, required.
- `updated_at`: timestamptz.

Constraints and indexes:

- Primary or unique key `vm_id, metric_type`.
- Index on `alerting`.
- Index on `latest_observed_at`.

### alerts

Fields:

- `id`: UUID or ULID primary key.
- `vm_id`: text, required.
- `metric_type`: enum, required.
- `threshold`: double precision, required.
- `latest_value`: double precision, required.
- `consecutive_breaches`: integer, required.
- `status`: enum, `ACTIVE`, `RESOLVED`.
- `started_at`: timestamptz.
- `resolved_at`: timestamptz, nullable.
- `updated_at`: timestamptz.

Constraints and indexes:

- Unique active alert on `vm_id, metric_type` where `status = 'ACTIVE'`.
- Index on `status, updated_at DESC`.
- Index on `vm_id, status`.

## 10. Processing Rules

- API validates batch size, sample shape, metric type, value bounds, and timestamps.
- API creates or reuses an ingestion batch when `Idempotency-Key` is present.
- API publishes accepted samples to Kafka and returns `202 Accepted` after Kafka acknowledgment.
- Worker consumes sample messages from Kafka.
- Worker inserts each raw sample into `metric_samples`.
- If `sample_id` already exists, the worker treats the sample as duplicate, increments duplicate count if batch tracking exists, and skips aggregate and alert updates.
- For newly inserted samples, the worker upserts hourly aggregates.
- For CPU and memory samples, the worker updates `vm_metric_health` and `alerts`.
- A VM metric becomes alerting when the latest 3 consecutive samples for that VM and metric are above the configured threshold.
- An active alert resolves when a newer CPU or memory sample is at or below threshold.
- Disk and network samples are stored and aggregated but do not produce alerts in the MVP.
- Out-of-order samples are stored. The MVP updates latest health only when the sample `observed_at` is newer than or equal to the current `latest_observed_at`; production can add a reconciliation job for late-arriving samples.
- Processing failures are retried by the worker. After retry exhaustion, production sends the message to a DLQ.

Async justification:

- 50k-100k samples/sec and p95 ingestion latency under 200ms require the API to avoid doing all raw inserts, aggregate updates, and alert calculations inline.
- Kafka provides buffering, backpressure handling, replay, and worker-based horizontal scaling.

## 11. Idempotency / Duplicate Handling

Duplicate handling is required because clients may retry ingestion requests.

Idempotency keys:

- Batch-level: optional `Idempotency-Key` header.
- Sample-level: required `sample_id`; fallback is `vm_id + metric_type + observed_at`.

First request behavior:

- API creates an ingestion batch, publishes samples to Kafka, and returns `202`.

Duplicate batch request behavior:

- If the same `Idempotency-Key` is seen again, return `200` with the existing `batch_id` and `idempotent_replay=true`.

Duplicate sample behavior:

- Worker attempts insert into `metric_samples`.
- Unique constraint prevents duplicate persistence.
- Duplicate samples do not update hourly aggregates, health, or alerts.

MVP choice:

- Use `200` for idempotent batch replay because it is client-friendly for retries.
- Use database uniqueness as the correctness boundary because Kafka may deliver duplicate messages.

## 12. Transaction and Concurrency Considerations

Transaction boundary:

- Worker processing of one sample should insert the raw sample, update the hourly aggregate, update health/alert state if applicable, and update batch counters in one database transaction when practical.
- If the raw insert conflicts on `sample_id`, the transaction should skip derived updates.

Concurrent risks:

- Duplicate messages may be processed by different workers.
- Multiple samples for the same VM and metric may arrive concurrently.
- Out-of-order samples may arrive after health was updated.
- Kafka publish may partially fail after an ingestion batch was created.

MVP handling:

- Unique constraints prevent duplicate raw samples and duplicate active alerts.
- Aggregate upserts use atomic `INSERT ... ON CONFLICT DO UPDATE`.
- Health updates should protect against older samples overwriting newer state by comparing `observed_at`.
- Batch status is best effort for the MVP; sample uniqueness remains the correctness boundary.

Production improvements:

- Transactional outbox if API acceptance and Kafka publish must be atomic.
- Retry topic and DLQ with explicit retry counts.
- Row-level locking or optimistic locking for strict per-VM health ordering.
- Reconciliation job for out-of-order samples.
- Partitioned raw tables and retention jobs.

## 13. Kafka and Worker Design

Kafka is implemented in the MVP because async processing, high throughput, retry, and buffering are explicit requirements.

Topics:

- `vm.metric-samples.v1`: primary sample ingestion topic.
- `vm.metric-samples.dlq.v1`: production DLQ topic.

Message schema:

```json
{
  "message_id": "msg-01JZ...",
  "batch_id": "batch-01JZ...",
  "sample_id": "sample-01JZ...",
  "vm_id": "vm-123",
  "metric_type": "cpu_usage_percent",
  "value": 91.5,
  "observed_at": "2026-06-21T21:00:00Z",
  "received_at": "2026-06-21T21:00:01Z",
  "request_id": "req-01JZ..."
}
```

Partition key:

- `vm_id`.

Consumer group:

- `vm-metrics-processors`.

Worker responsibilities:

- Consume sample messages.
- Validate message shape defensively.
- Insert raw samples idempotently.
- Update hourly aggregates.
- Update CPU and memory health.
- Create or resolve alerts.
- Update ingestion batch counters.
- Log failures with request ID, batch ID, sample ID, and VM ID.

Retry behavior:

- MVP: let Kafka redeliver on worker failure; keep processing idempotent.
- Production: track retry count, use retry topics/backoff, and send exhausted messages to DLQ.

Metrics to emit:

- Messages consumed.
- Processing latency.
- Duplicate samples.
- Insert failures.
- Aggregate update failures.
- Alert transitions.
- Retry count.
- DLQ count.
- Consumer lag.

## 14. Validation Rules

API request validation:

- `samples` is required and must contain 1 to `MAX_INGEST_BATCH_SIZE` items.
- Default `MAX_INGEST_BATCH_SIZE`: 1,000.
- `sample_id` is required, non-empty, and bounded length.
- `vm_id` is required, non-empty, and bounded length.
- `metric_type` must be one of:
  - `cpu_usage_percent`
  - `memory_usage_percent`
  - `disk_usage_percent`
  - `network_in_bytes`
  - `network_out_bytes`
- `value` is required numeric.
- Percent metrics must be `0 <= value <= 100`.
- Network byte metrics must be `value >= 0`.
- `observed_at` is required and must be a valid timestamp.
- Reject timestamps too far in the future, default more than 5 minutes ahead.
- Query `start` must be before `end`.
- Raw metric query time range defaults to max 24 hours for MVP unless overridden by config.
- Pagination `limit` must be between 1 and `MAX_PAGE_SIZE`.

Implementation guidance:

- Use Pydantic models at the FastAPI API boundary.
- Keep domain-specific processing checks in service-layer functions so they are unit testable.

## 15. Error Handling

Use one JSON error shape:

```json
{
  "error": {
    "code": "string",
    "message": "string",
    "request_id": "string",
    "details": {}
  }
}
```

Common status codes:

- `400 bad_request`: invalid batch size, invalid time range, unsupported query combination.
- `404 not_found`: batch or VM health not found.
- `409 conflict`: reserved for non-idempotent conflicts if introduced later.
- `422 validation_error`: Pydantic request validation failure.
- `500 internal_server_error`: unexpected server error.
- `503 dependency_unavailable`: PostgreSQL or Kafka unavailable.

Errors must not expose stack traces, credentials, broker addresses with secrets, or internal SQL details.

## 20. Scalability Considerations

Initial version scales through:

- Stateless FastAPI replicas.
- App Platform ingress/load balancing.
- Batched ingestion.
- Kafka buffering and partitioning.
- App Platform worker replicas in one consumer group.
- PostgreSQL indexes for query paths.
- Bounded request payloads.
- Bounded time-range queries.
- Pagination.
- Hourly aggregate table for longer-term summaries.

Kafka is required because:

- Processing can be eventually consistent.
- The API must stay below 200ms p95.
- Peak rate is 100k samples/sec.
- Retry and DLQ behavior are required.

Expected bottlenecks:

- PostgreSQL write throughput for raw samples and aggregates.
- Hot VMs if Kafka partitioning or DB row updates concentrate load.
- Aggregate upsert contention for high-frequency VM/metric/hour rows.
- Worker lag if DB writes cannot keep up.

Production scale improvements:

- Increase Kafka partitions and worker replicas.
- Use producer batching and compression.
- Use PostgreSQL time partitioning and efficient batch inserts.
- Drop old raw partitions for 7-day retention.
- Drop old aggregate partitions for 90-day retention.
- Consider TimescaleDB or a dedicated time-series store only if PostgreSQL cannot meet real load, but keep this outside the required MVP unless allowed.
- Add transactional outbox if acceptance tracking must be exactly coordinated with Kafka publish.
- Add Redis/Valkey for rate limiting or hot query caching if needed.
- Add load/performance tests to validate throughput.

## 22. Trade-offs

- Async Kafka processing is selected for the MVP because the problem explicitly requires asynchronous processing, high ingestion throughput, retry behavior, and production Managed Kafka.
- Batch ingestion is required to make request rate manageable and to meet the p95 latency target.
- PostgreSQL is the queryable durable source of truth because it stores raw samples, aggregates, health, and alerts with uniqueness constraints.
- Kafka is a durable ingestion buffer, not the query source of truth.
- The MVP prioritizes idempotent worker processing over exactly-once Kafka semantics because Kafka consumers can receive duplicate messages.
- The API returns `202` before processing completes because processing can be eventually consistent within 30 seconds.
- Alert state is updated by workers rather than inline in the API to keep ingestion fast.
- Hourly aggregates are stored because querying 7 days of raw high-volume data directly is expensive.
- The MVP includes only global CPU and memory thresholds to keep the implementation focused.
- The design focuses on a production-candidate slice rather than a half-finished distributed system: validation, persistence, idempotency, bounded queries, worker processing, Docker, tests, and deployment readiness.
